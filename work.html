Automated Financial Data Pipeline

Data Engineering ETL Python
8
DQ Checks Per Run

Before

Two financial data sources with inconsistent formats, mismatched currency codes, and different ID schemes. Manual consolidation in Excel: error-prone, no audit trail, no quality checks. Duplicate transaction IDs from card-processor retry logic and null merchant names were causing formula errors in downstream reports.

After

  • Multi-source extraction: two CSV feeds + simulated FX-rate API merged cleanly
  • 12-step deterministic cleaning: dedup, null fills, type coercion, currency validation
  • 8 automated DQ checks on every run: null, duplicate, schema, amount range, z-score anomaly, date range, category, and status validation
  • All amounts normalised to USD via configurable FX rates
  • Full audit trail: per-run JSON DQ report + pipeline_runs database record
  • MySQL output connects directly to Metabase, Tableau, or Grafana

What I Shipped

A production-style Python ETL pipeline that extracts from two CSV sources and a simulated FX-rate API, applies a 12-step cleaning process, runs 8 data-quality validations, converts all amounts to USD, and loads results into MySQL via idempotent batch upsert. Every run writes a structured JSON DQ report and a pipeline_runs audit record. Includes dry-run mode (no database required), 35 pytest tests across extract, transform, and validate layers, and 10 analytics-ready SQL queries for BI tools.

Tech: Python · MySQL · pandas · pytest · ETL · Data Quality Automation

What I’d tell myself before starting

The cleaning step took longer than the pipeline itself. Twelve steps sounds like a lot until you see what raw data from two mismatched systems actually looks like. Schema standardisation first, automation second — that order matters.

Ford Stock Return Prediction

R Statistical Modeling Time-Series
4
Models Compared

Before

Ford’s monthly stock return is shaped by dozens of macroeconomic variables — S&P 500 movements, oil prices, treasury yields, VIX, exchange rates. Standard linear regression doesn’t handle multicollinearity or time-structure in financial data well. 263 monthly observations (Jan 2004–Nov 2025) spanning multiple market cycles.

After

  • 4 regression models built and compared: OLS, Stepwise AIC, Mallows’ Cp, CAPM
  • Time-aware train/test split to respect temporal ordering
  • Robust HC standard errors applied to correct for heteroskedasticity
  • VIF diagnostics used to identify and address multicollinearity
  • Significant macro predictors of Ford’s monthly return identified

What I Shipped

A full regression study in R comparing four approaches to predicting Ford monthly returns from macroeconomic and market data. Includes model selection, multicollinearity diagnostics, heteroskedasticity-robust inference, and out-of-sample evaluation using a time-aware split.

Tech: R · ggplot2 · dplyr · MASS · leaps · lmtest · estimatr · corrplot · knitr

What I’d tell myself before starting

Running stepwise selection without checking multicollinearity first gives you a confident model with inflated coefficients. VIF diagnostics before model selection, not after.

Market Returns Around VIX Peaks

Python Event Study Finance
~40
VIX Peaks Analyzed

Before

VIX spikes are widely treated as fear signals, but the equity return pattern around peaks — before and after — is rarely examined systematically. Do markets actually mean-revert post-spike? Does the response differ by sector or volatility tier? No structured answer existed for the 2016–present period.

After

  • ~40 VIX peaks detected (2016–present, local max ≥25, ±5-day window)
  • S&P 500 return: −0.24%/day pre-peak → +0.19%/day post-peak
  • High-volatility stocks: −0.29% → +0.30% (amplified reversal)
  • Energy and Financials led post-peak recovery across GICS sectors
  • Event windows: ±10 trading days around each peak

What I Shipped

A Python-based event study examining equity market behavior around CBOE VIX spikes. Automated peak detection, average abnormal return (AAR) and cumulative abnormal return (CAR) calculations, volatility-sorted portfolio analysis, and GICS sector breakdown. Visualizations for all four dimensions.

Tech: Python · pandas · numpy · scipy · yfinance · matplotlib · Jupyter

What I’d tell myself before starting

Sector labels from live API data shift over time as companies reclassify. The size proxy using volatility sorting is imperfect but captures the intended risk-tier effect well enough for this event window.

Asian Option Pricing with Neural Networks

PyTorch Quantitative Finance Deep Learning
5
Architectures Compared

Before

Pricing arithmetic Asian options requires Monte Carlo simulation with tens of thousands of paths. Computationally expensive by design, with no closed-form solution. The research question: can a neural network trained on synthetic GBM data serve as a fast, accurate surrogate pricer?

After

  • MLP best performer: MAE 0.0256, RMSE 0.0860
  • Inference near-instant vs. 50,000-path Monte Carlo simulation
  • CNN, LSTM, CNN-LSTM, and TCN all tested — all overfit on available data
  • Adaptive sampling did not improve generalization
  • Dataset size identified as the binding constraint, not architecture

What I Shipped

A PyTorch-based surrogate pricing model trained on synthetic option data generated via Geometric Brownian Motion simulation. Five architectures benchmarked (MLP, CNN, LSTM, CNN-LSTM, TCN) with full training curves. Dataset construction, adaptive sampling experiments, and out-of-sample evaluation all documented.

Tech: Python · PyTorch · numpy · scipy · pandas · Geometric Brownian Motion · Jupyter

What I’d tell myself before starting

MLP generalizes better here because it has fewer parameters to overfit with. The more complex architectures need data volume that the simulation budget didn’t reach. More data > better architecture when you’re data-constrained.

What I Shipped
